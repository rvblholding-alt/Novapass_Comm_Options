Comparatif architectures de communication — Mars 2026
Architecture 01
VPN + NanoPi
Routeur intermédiaire / Machine dédiée
Brique hardware sur chaque site gérant le tunnel VPN. Le Master communique avec le backend via HTTP REST classique à travers le tunnel.
Architecture 02
WireGuard VPN
Tunnel VPN logiciel / Peer-to-peer
Protocole VPN intégré dans le firmware du Master. Pas de hardware intermédiaire. Le Master communique avec le backend via HTTP REST classique à travers le tunnel.
Architecture 03
WebSocket + mTLS
WSS / TLS 1.3 / Certificat X.509 par device / Architecture événementielle
Connexion persistante bidirectionnelle initiée par le Master vers ComplyStation. Certificat gravé en usine. Impose une architecture event-driven côté backend.

Architecture Master → Backend
Modèle de communication applicatif
REST / MVC classique
Le Master envoie des appels HTTP POST standards. Architecture connue, facile à tester et à déboguer.
REST / MVC classique
Idem VPN + NanoPi. Le tunnel est transparent pour l'application. Aucun changement de paradigme applicatif côté backend.
Mais : pour les commandes cloud → Master (ex. ouverture de porte), le REST pur n'est pas envisageable. Il faudrait quand même un mécanisme de push (SignalR ou WebSocket) pour ce flux descendant, en parallèle du REST montant.
Event-Driven — nouveau paradigme
Impose une architecture événementielle entre le Master et le backend. Tous les messages arrivent sur un seul canal persistant : il faut parser chaque message pour savoir s'il s'agit d'une alarme, d'un log, d'un acquittement, d'un état de porte.
Le routage automatique par URL (REST) disparaît — le routage des messages devient manuel dans le code. L'asynchronisme complet augmente la complexité et les risques de bugs. Une partie de la complexité réseau est déplacée dans l'applicatif.
Overhead réseau
Volume de données, connexions/déconnexions
Lourd
Chaque événement = une nouvelle connexion HTTP complète. Pour un simple JSON de 100 octets, le overhead HTTP (headers, handshake TLS) représente plusieurs dizaines de kilooctets. Très lourd à l'échelle.
Lourd
Même problème que VPN + NanoPi côté applicatif : le REST implique de nombreuses connexions/déconnexions pour les événements et changements d'état fréquents.
Minimal
Connexion persistante établie une seule fois. Les messages suivants sont de quelques octets de framing. Idéal pour les échanges fréquents d'états et d'événements.
Maîtrise technologique

Maîtrisé

Maîtrisé
Côté applicatif, identique à une API REST standard. La nouveauté est limitée à l'intégration de WireGuard dans le firmware.
A acquérir
L'architecture event-driven sur WebSocket n'est pas encore maîtrisée par l'équipe. Courbe d'apprentissage réelle, risque de bugs accru en phase initiale.
Ce n'est pas une impasse — c'est une technologie mature et bien documentée. Mais le coût d'apprentissage est un facteur de risque projet à intégrer dans le planning. La complexité est surtout en conception initiale ; la maintenance récurrente est ensuite comparable à du REST.
Rôle de SignalR
Backend → Frontend dashboard & Backend → Master
Dans les trois architectures, la communication Backend → Frontend (dashboard ComplyStation dans le navigateur) se fera via SignalR — technologie Microsoft éprouvée pour le push temps réel vers le navigateur. Ce segment est identique quelle que soit l'architecture choisie côté Master.

Point soulevé par Daniel (Dinec) : SignalR supporte nativement WebSocket comme transport sous-jacent et gère déjà une bonne partie de l'aiguillage des messages. Si SignalR est utilisé pour le flux Backend → Frontend, il serait logique d'explorer s'il peut également prendre en charge le flux Master → Backend, évitant ainsi de maintenir deux paradigmes de communication distincts. À valider avec l'équipe ComplyStation.

Conclusion
Quelle que soit l'architecture choisie
La vidéosurveillance emprunte toujours un canal séparé — indépendant de l'architecture de contrôle d'accès retenue.
Aucune des trois architectures n'est adaptée au transport de flux vidéo. Ce n'est pas un critère de différenciation entre elles : dans tous les cas, la vidéosurveillance nécessite son propre canal de communication, déterminé par le matériel vidéo choisi.

Le protocole vidéo dépend du fabricant et du cas d'usage :
— RTSP : standard historique des caméras IP (Axis, Hanwha, Hikvision, Bosch). Flux direct vers un NVR ou vers le cloud.
— WebRTC : visualisation temps réel dans un navigateur. Approche des acteurs cloud-native (Verkada, Avigilon Alta).
— HLS / DASH : streaming quasi-temps réel via HTTP standard, plus simple à faire passer les firewalls.

Les caméras IP modernes sont des équipements Linux embarqués alimentés en PoE qui envoient leur flux directement vers le cloud ou un NVR, sans passer par le contrôleur d'accès. La tendance du marché va vers la suppression du NVR au profit du stockage embarqué sur la caméra — dans le même mouvement cloud-native que NovaPass pour le contrôle d'accès.

Si la vidéosurveillance entre dans le scope NovaPass, c'est un chantier architectural distinct à traiter séparément, dont le protocole sera dicté par le choix du matériel vidéo retenu.
Architecture 01
VPN + NanoPi

Pertinent pour des environnements très conservateurs exigeant une séparation matérielle OT/IT. Architecture REST familière côté développement. Difficile à justifier à grande échelle en raison du coût hardware et de la complexité terrain. De plus, un mécanisme de push complémentaire reste nécessaire pour les commandes cloud → Master.

Architecture 02
WireGuard VPN

Option très crédible : cryptographie solide, pas de hardware intermédiaire, architecture REST conservée côté applicatif. Effort de développement nettement plus faible que WebSocket en phase initiale. Risques principaux : dépendance à UDP dans les réseaux filtrés, et nécessité d'un mécanisme push complémentaire pour les commandes descendantes.

Architecture 03 — ARCHITECTURE RETENUE
WebSocket + mTLS

Option la plus cohérente pour un produit cloud plug & play avec supervision native et overhead réseau minimal. Résout nativement le problème des commandes descendantes (cloud → Master) sans mécanisme additionnel. Impose un changement de paradigme applicatif (event-driven) et une infrastructure PKI sérieuse — deux risques projet réels à anticiper, pas des obstacles définitifs. SignalR peut significativement réduire la complexité d'aiguillage.
